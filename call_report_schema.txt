================================================================================
 DOCUMENTACIÓN DEL REPORTE DE LLAMADA — SarahIA
 Para uso del desarrollador Frontend
================================================================================

El backend envía este JSON al frontend cada vez que hay una actualización
significativa del estado de la llamada. Se envía como mínimo dos veces:
  1. Cuando finaliza la conversación (post_call_transcription).
  2. Cuando llega el audio (post_call_audio) — incluye "base64_audio".

El frontend debe sobrescribir el estado anterior usando "call_id" como clave.

================================================================================
 NOTAS IMPORTANTES PARA EL FRONTEND
================================================================================

1. EL REPORTE SE ENVÍA VARIAS VECES
   El mismo call_id puede recibirse múltiples veces con datos progresivamente
   más completos. Orden habitual:
     a) Al iniciar el reintento con número alternativo (status: RETRYING).
     b) Al finalizar la conversación (sin audio aún, base64_audio: null).
     c) Al llegar el audio (base64_audio presente).
   → Siempre sobrescribir el estado anterior usando call_id como clave.

2. CÓMO DETERMINAR EL RESULTADO FINAL
   - Estado global del ciclo:  .status
   - Decisión del trabajador:  .call_context.confirmation
   - Resultado IA del último intento:
       .call_record[.call_record.last_called].elevenlabs_analysis.was_success
   El campo .status es el más fiable para el estado del ciclo completo.

3. MÚLTIPLES INTENTOS
   Si se intentaron varios números, call_record tendrá entradas para cada uno:
     { "phone": {...}, "alternative_phone": {...}, "alternative_phone_2": {...} }
   Usar "last_called" para identificar cuál fue el definitivo.

4. AUDIO BASE64
   - Llega en un segundo envío del reporte.
   - Si base64_audio es null o el campo no existe, el audio aún no está disponible.

5. CAMPOS VACÍOS VS NULL
   - "" (cadena vacía): el campo existía pero no tenía valor (ej: hourly_rate no informada).
   - null: el campo no aplicó o no fue posible obtenerlo.
   - Ambos casos deben tratarse como "sin dato" en la UI.
